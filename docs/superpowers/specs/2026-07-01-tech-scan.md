## Next.js 16 — breaking changes that shape the plan

1. **`params` and `searchParams` are `Promise<...>` — must `await`.**
   ```tsx
   export default async function Page(
     { params }: { params: Promise<{ slug: string }> }
   ) { const { slug } = await params }
   ```
   In a client component use `use(params)` instead. (dm1 already does the async
   pattern correctly — confirmed in its `[id]/page.tsx`.)

2. **`searchParams` opts a page into dynamic rendering.** For pagination use a
   route segment `page/[n]`, NOT `?page=`. Our QA-bar example selection should
   therefore be a **route/path or client-side state**, not a `searchParams` read
   on statically-rendered pages (or those pages go dynamic — acceptable if we
   choose that, but must be deliberate).

3. **`generateStaticParams`** returns plain objects (not Promises); parent params
   passed into a child's `generateStaticParams` are synchronous.

4. **`output: 'export'` (full static) forbids** dynamic routes without
   `generateStaticParams`, `dynamicParams: true`, `cookies()`/`headers()`,
   `searchParams`, server actions, ISR. **Decision:** do NOT use `output:'export'`.
   dm1 doesn't either — deploy Vercel-native so detail routes render on demand.

5. **The scale decision (learned from dm1):** dm1 generates ~300k participants and
   deliberately does **not** `generateStaticParams` the profile pages — pre-rendering
   hundreds of thousands of pages at build is infeasible. **Our plan:** pre-render
   list pages + a bounded set; render individual profile/detail routes **on demand**
   (default dynamic). This keeps builds fast and Vercel deploys clean.

- `@fontsource-variable/inter` self-hosted works (matches extension; avoids remote

## Client-side XLSX export — USE `exceljs` (matches extension exactly)

**Decision (supersedes SheetJS):** the extension itself serializes XLSX with
**`exceljs@4.4.0`** (`captello-envoy/src/sidepanel/lib/export.ts`). To make the
demo's export byte-compatible with the extension's, the demo uses the SAME lib
and SAME shape. No CDN/registry caveat — `exceljs` is a normal npm package and
works in the browser (`await wb.xlsx.writeBuffer()` → Blob → download).

**Exact contract to replicate (from the extension's `export.ts`):**
- **CSV** — one file PER TYPE: `envoy-individuals-<ts>.csv`,
  `envoy-companies-<ts>.csv`. Header row = the `label`s of `INDIVIDUAL_FIELDS` /
  `COMPANY_FIELDS`. `csvEscape`: wrap in quotes and double internal quotes if the
  value contains `,` `"` or newline. `types` array → titleCased, `", "`-joined.
  `additionalInfo` → `JSON.stringify(value ?? {})`. null/undefined → `""`.
  A type with zero records is skipped (no empty file).
- **XLSX** — ONE workbook, two worksheets `"Individuals"` and `"Companies"`
  (both always present, header-only if empty). Row 1 = labels, bold.
  Filename `envoy-results-<ts>.xlsx`. `<ts>` = `new Date().toISOString()` with
  `[:.]` replaced by `-`.
- **JSON** — raw ground-truth array(s) (demo-only convenience, no extension
  equivalent).

CSV + JSON are hand-rolled; XLSX via `exceljs`. Field key/label/order copied
verbatim from the extension (22 individual fields, 20 company fields — see the
design spec §5 and the verbatim list in the plan).
